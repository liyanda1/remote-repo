# 部门 TDD 开发流程主入口

**使用时机：** 开始任何开发任务时首先调用此 skill

**流程概览：**
```
需求输入 → 详设文档 → 用例设计 → TDD 编码（红-绿-重构）
```

---

## 阶段检测与路由

根据项目当前状态，自动路由到正确的阶段：

### 1. 检查当前项目状态

检查以下文件/目录判断当前阶段：

| 检查项 | 存在？ | 路由到 |
|--------|--------|--------|
| `docs/design/*.md` 详设文档 | 不存在 | **tdd-design**（详设阶段） |
| `docs/test-cases/*.md` 用例文档 | 不存在 | **tdd-test-design**（用例设计阶段） |
| `src/` 或项目代码目录 | 不存在/不完整 | **tdd-coding**（TDD 编码阶段） |
| 以上全部存在 | 全部存在 | 询问用户新需求或维护现有功能 |

### 2. 路由决策流程

```
会话开始
    │
    ├─→ 是否有需求变更/新需求？
    │   ├─→ 是 → 进入详设阶段 (tdd-design)
    │   └─→ 否 → 继续检查
    │
    ├─→ 详设文档是否已批准？
    │   ├─→ 否 → 继续详设阶段 (tdd-design)
    │   └─→ 是 → 继续检查
    │
    ├─→ 用例设计是否完成？
    │   ├─→ 否 → 进入用例设计阶段 (tdd-test-design)
    │   └─→ 是 → 继续检查
    │
    └─→ 进入 TDD 编码阶段 (tdd-coding)
```

---

## 各阶段说明

### 阶段一：详设阶段 (tdd-design)

**产出物：** `docs/design/YYYY-MM-DD-<feature-name>-design.md`

**主要内容：**
- 需求背景与目标
- 接口契约定义（输入/输出/异常）
- 核心算法/逻辑伪代码
- 边界条件与错误处理
- 测试场景清单（Test Inventory）

**模板位置：** 使用 `templates/detail-design-template.md`（可自定义）

---

### 阶段二：用例设计阶段 (tdd-test-design)

**产出物：** `docs/test-cases/YYYY-MM-DD-<feature-name>-cases.md`

**主要内容：**
- 基于详设的测试用例设计
- 测试场景覆盖：
  - 正常路径（Happy Path）
  - 错误处理（Error Handling）
  - 边界条件（Boundary/Edge）
  - 安全相关（Security，如适用）
- 测试数据准备
- 预期结果定义

**规则：**
- 负面测试比例 >= 40%
- 每个详设中的验证点必须对应至少一个测试用例

---

### 阶段三：TDD 编码阶段 (tdd-coding)

**遵循严格的 TDD 红-绿-重构循环：**

#### Step 1: TDD Red — 编写失败的测试
- 基于用例设计编写测试代码
- **所有测试必须先失败**（验证测试有效性）
- 测试覆盖：单元测试 + 集成测试（如适用）

#### Step 2: TDD Green — 最简实现
- 编写最少代码使测试通过
- 不允许提前实现未测试的功能
- 不允许过度设计

#### Step 3: TDD Refactor — 重构优化
- 在测试保持通过的前提下重构
- 提升代码质量、可读性、可维护性
- 每次重构后必须运行全部测试

**产出物：**
- 测试代码：`tests/test_*.py`（或其他语言对应目录）
- 实现代码：`src/` 目录下对应模块

---

## 关键原则

### 铁律
```
不允许在没有失败测试的情况下编写实现代码
```

### 质量门禁

| 门禁项 | 阈值 | 说明 |
|--------|------|------|
| 行覆盖率 | ≥ 80% | 核心逻辑必须覆盖 |
| 分支覆盖率 | ≥ 70% | 条件分支必须覆盖 |
| 负面测试比例 | ≥ 40% | 错误/边界场景覆盖 |

### 文档规范

1. **详设文档**必须包含：
   - 接口契约（输入/输出/异常）
   - 核心算法伪代码
   - 边界条件矩阵
   - 测试场景清单

2. **用例文档**必须包含：
   - 测试场景编号与详设对应关系
   - 测试数据（具体值，非模糊描述）
   - 预期结果（可验证的具体断言）

3. **测试代码**必须包含：
   - 测试函数与用例文档的对应注释
   - 测试分层标记（#[unit] / #[integration]）

## 使用方式

### 启动开发流程

```
> 我要开发一个新功能：[功能描述]
> 请使用 tdd-workflow skill
```

### 继续已有任务

```
> 继续开发 [功能名]
> 请使用 tdd-workflow skill
```

### 自定义模板

如需使用自定义详设模板：

```
> 使用自定义详设模板：templates/my-design-template.md
> 请使用 tdd-workflow skill
```

## 项目结构规范

```
project/
├── docs/
│   ├── design/                    # 详设文档目录
│   │   └── YYYY-MM-DD-xxx-design.md
│   └── test-cases/                # 用例文档目录
│       └── YYYY-MM-DD-xxx-cases.md
├── src/                           # 源代码目录
│   └── ...
├── tests/                         # 测试代码目录
│   └── ...
└── templates/                     # 模板目录（可选）
    └── detail-design-template.md  # 自定义详设模板
```

## 集成关系

**调用：** 用户直接调用
**路由到：** 
- `tdd-design`（详设阶段）
- `tdd-test-design`（用例设计阶段）
- `tdd-coding`（TDD 编码阶段）

**关键文件：**
- `docs/design/*.md` — 详设文档
- `docs/test-cases/*.md` — 用例文档
- `src/` — 实现代码
- `tests/` — 测试代码
