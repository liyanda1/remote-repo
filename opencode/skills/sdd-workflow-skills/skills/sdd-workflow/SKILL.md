# 部门 SDD 开发流程主入口

**使用时机：** 开始任何开发任务时首先调用此 skill

**流程概览：**
```
需求输入 → 组件级设计（高层）→ SR/AR 级设计（低层）→ 用例设计 → 编码实现 → 测试验证
```

---

## 阶段检测与路由

根据项目当前状态，自动路由到正确的阶段：

### 1. 检查当前项目状态

检查以下文件/目录判断当前阶段：

| 检查项 | 存在？ | 路由到 |
|--------|--------|--------|
| `docs/design/*-component-design.md` 组件级设计 | 不存在 | **sdd-design**（组件级设计阶段） |
| `docs/design/SR*-requirement-design.md` SR 需求级设计 | 不存在 | **sdd-design**（SR 需求级设计阶段） |
| `docs/design/AR*-requirement-design.md` AR 需求级设计 | 不存在 | **sdd-design**（AR 需求级设计阶段） |
| `docs/test-cases/*.md` 用例文档 | 不存在 | **sdd-test-design**（用例设计阶段） |
| `src/` 或项目代码目录 | 不存在/不完整 | **sdd-coding**（编码阶段） |
| 以上全部存在 | 全部存在 | 询问用户新需求或维护现有功能 |

### 2. 路由决策流程

```
会话开始
    │
    ├─→ 是否有新组件/模块需求？
    │   ├─→ 是 → 进入组件级设计阶段 (sdd-design)
    │   └─→ 否 → 继续检查
    │
    ├─→ 组件级设计是否已完成？
    │   ├─→ 否 → 继续组件级设计阶段 (sdd-design)
    │   └─→ 是 → 继续检查
    │
    ├─→ 是否有 SR/AR 级需求需要细化？
    │   ├─→ 是 → 进入 SR/AR 需求级设计阶段 (sdd-design)
    │   └─→ 否 → 继续检查
    │
    ├─→ 用例设计是否完成？
    │   ├─→ 否 → 进入用例设计阶段 (sdd-test-design)
    │   └─→ 是 → 继续检查
    │
    └─→ 进入编码阶段 (sdd-coding)
```

## 各阶段说明

### 阶段一：组件级设计（高层设计）

**产出物：** `docs/design/<组件名>-component-design.md`

**主要内容：**
- 组件背景与目标
- 组件架构设计
- 子模块划分与职责
- 对外接口契约定义
- 与外部组件的交互
- 测试策略

**设计层级：** 组件级（Component Level）
- 关注组件整体架构
- 定义子模块边界和交互
- 不涉及具体类/函数实现

---

### 阶段二：SR/AR 需求级设计（低层设计）

**产出物：**
- SR 级：`docs/design/<SR单号>-<需求名>-requirement-design.md`
- AR 级：`docs/design/<AR单号>-<需求名>-requirement-design.md`

**主要内容：**
- 需求实现方案
- 类/结构体设计
- 详细接口定义（API 级别）
- 算法/状态机设计
- 数据结构定义
- 测试场景清单

**设计层级：** 需求级（Requirement Level）
- 关注具体需求实现
- 定义具体类、接口、算法
- 为编码提供直接指导

---

### 阶段三：用例设计阶段

**产出物：** `docs/test-cases/<AR单号>-<需求名>-cases.md`

**主要内容：**
- 基于详设的测试用例设计
- 测试场景覆盖：
  - 正常路径（Happy Path）
  - 错误处理（Error Handling）
  - 边界条件（Boundary/Edge）
  - 性能/压力测试（如适用）
- 测试数据准备
- 预期结果定义

**规则：**
- 负面测试比例 >= 40%
- 每个详设中的验证点必须对应至少一个测试用例

---

### 阶段四：编码实现阶段

**遵循设计驱动开发：**

#### Step 1: 代码框架生成
- 基于组件详设生成代码框架
- 定义头文件、接口、类结构
- 建立项目目录结构

#### Step 2: 测试先行
- 基于用例设计编写测试代码
- 所有测试必须先失败（验证测试有效性）
- 测试覆盖：单元测试 + 集成测试（如适用）

#### Step 3: 功能实现
- 基于设计文档实现功能
- 确保实现与设计一致
- 不允许偏离设计文档

#### Step 4: 重构优化
- 在测试保持通过的前提下重构
- 提升代码质量、可读性、可维护性
- 每次重构后必须运行全部测试

**产出物：**
- 测试代码：`tests/test_*.cpp`（或其他语言对应目录）
- 实现代码：`src/` 目录下对应模块

## 关键原则

### 铁律
```
设计文档是开发的唯一依据
不允许在没有设计文档的情况下开始编码
```

### 质量门禁

| 门禁项 | 阈值 | 说明 |
|--------|------|------|
| 行覆盖率 | ≥ 80% | 核心逻辑必须覆盖 |
| 分支覆盖率 | ≥ 70% | 条件分支必须覆盖 |
| 负面测试比例 | ≥ 40% | 错误/边界场景覆盖 |

### 文档规范

1. **组件级设计文档**必须包含：
   - 组件架构图
   - 子模块划分与职责
   - 对外接口契约
   - 与外部组件交互图
   - 测试策略

2. **需求级设计文档**必须包含：
   - 类图/结构体定义
   - 详细接口定义（参数、返回值、异常）
   - 核心算法伪代码
   - 状态机设计
   - 测试场景清单

3. **用例文档**必须包含：
   - 测试场景编号与详设对应关系
   - 测试数据（具体值，非模糊描述）
   - 预期结果（可验证的具体断言）

4. **测试代码**必须包含：
   - 测试函数与用例文档的对应注释
   - 测试分层标记（#[unit] / #[integration]）

## 使用方式

### 启动开发流程

```
> 我要开发一个新功能：[功能描述]
> 请使用 sdd-workflow skill
```

### 继续已有任务

```
> 继续开发 [功能名]
> 请使用 sdd-workflow skill
```

### 自定义模板

如需使用自定义详设模板：

```
> 使用自定义详设模板：templates/requirement-design-template.md
> 请使用 sdd-workflow skill
```

## 项目结构规范

```
project/
├── docs/
│   ├── design/                              # 设计文档目录
│   │   ├── <组件名>-component-design.md          # 组件级设计（高层）
│   │   ├── <SR单号>-<需求名>-requirement-design.md   # SR 需求级设计（低层）
│   │   └── <AR单号>-<需求名>-requirement-design.md   # AR 需求级设计（低层）
│   └── test-cases/                          # 用例文档目录
│       └── <SR/AR单号>-<需求名>-cases.md
├── src/                                     # 源代码目录
│   └── ...
├── tests/                                   # 测试代码目录
│   └── ...
└── templates/                               # 模板目录（可选）
    ├── component-design-template.md         # 组件级设计模板（高层）
    └── requirement-design-template.md       # 需求级设计模板（低层）
```

## 集成关系

**调用：** 用户直接调用
**路由到：** 
- `sdd-design`（设计阶段）
- `sdd-test-design`（用例设计阶段）
- `sdd-coding`（编码阶段）

**关键文件：**
- `docs/design/*-component-design.md` — 组件级设计文档（高层）
- `docs/design/*-requirement-design.md` — SR/AR 需求级设计文档（低层）
- `docs/test-cases/*.md` — 用例文档
- `src/` — 实现代码
- `tests/` — 测试代码
